成为一名git大师

我前段时间学习了Bitbucket（类似GitHub的git仓库托管服务）推出的Git Tutorials系列文章，收获很大。记得我在2016年初，由于忍受不了公司缓慢的网速导致使用svn进行项目代码同步时间太长，于是就劝说原单位的项目负责人使用git替代svn。当时我罗列了一堆git做为项目版本控制程序的优点，但是我还没有太多关于git的使用经验，只是基本命令的使用，并没有系统的学习。而看了一些专业的git书籍后，可能是由于内容繁多和深奥，我也没有坚持下来。如果能有一些简单上手的实际操作教程的话，或许能帮我提高git水平。Git Tutorials正好是一个图文并茂的教程，不仅能清晰明了地解释git工作原理，而且也给出了很多深入的操作技巧，引领初学者一步步成为一位git大师。

这篇博客意在记录学习git的心得体会、思考，以及整理出一份git进阶使用指南。

什么是git

git是一种追踪文件改动的管理程序。起初是一位程序天才Linus Torvalds为了使用免费的版本管理软件来在社区上维护自己开发的Linux系统的更新迭代，自己花两周写了一个程序，到后来成了影响至今的git。

git和svn的区别

同样都是版本控制软件，在多人协作方面，svn采用的是中心式架构，即使用一台服务器提供数据存储服务，团队所有人需要从那里获取并提交开发进度，同步的过程对服务器和网络环境非常依赖。

git采用去中心化的分布式架构，没有中央服务器的存在，团队中每个人都持有完整的项目拷贝，或者说每个人都可以扮演“服务器”的角色。所以对于分支的操作大多可以在本地进行，减轻了对网络的依赖（除非需要远程仓库），使得项目管理更加高效、轻量、灵活。

三大区域

git仓库包含了3个重要的区域：

  1. 工作区working area：追踪当下文件的改动
  2. 缓冲区staging area：保存改动前，需要把改动添加到一个临时的暂存区域，可以理解为buffer，在此处的改动也可以随时撤销和更新
  3. 提交历史区commiting history：通过保存快照的方式，记录每次改动而组成的历史时间线

分离状态

git使用一个HEAD指针来指向当前工作中最新的位置，就好比时间隧道中的“现在”/“当下”/“此时此刻”这个时间节点。当然你也可以将它指向到之前的某个提交历史，就好比回到了“过去”某个时间节点，这是一种被称为Detached HEAD的指针分离的状态，意思是在该状态下的指针位置与当前项目开发的分支环境已经分离。通常发生在“回到过去”(git checkout <commit>)的时候。

回到过去

这里是讲回到过去历史中的某个提交节点。通过命令git checkout <commit_id>/<tag_name>来实现。回到过去以后，当前的工作环境就恢复成了过去那个时刻的样子，而且并不会影响现在的环境。你可以随意的浏览、查看当时项目的状态，也可以进行改动，甚至在此基础上进行新的提交。

需要注意的是，穿越到过去历史的行为会造成上面描述的分离状态Detached HEAD。在该状态下，虽然可以允许改动或提交，但是这些提交并不属于任何一个分支。所以即使做了改动，这些改动后的提交也是遗孤Orphaned，而且在通过指针操作回到现在/当下/最新进度的时候就正式被遗弃了，它们的命运就是等着被垃圾回收给清理掉，好像从来没有存在过一样（当然不会立即清理，默认保留30天左右）。如果想保留这些提交的话，就需要在回到过去的提交节点上建立一个新的分支来做为它们新的归属，自然就不会被遗弃了。

穿越到了过去，做了该做的事，如何回到现在呢？通过命令git checkout <branch_name>即可回到某个分支当前的最新进度。

撤销改动

在正式提交前，对于文件中所做的改动“后悔”了怎么办？

  • 如果改动已经提到了缓冲区的话，就用git reset <file>把某个文件的改动撤回到工作区
  • 如果想撤销在工作区的改动，使用git checkout -- <file>撤销对某个文件的改动，即回到上次提交时候的状态
    • 命令中使用了--是为了将后面的参数转义成文件名来处理，毕竟checkout命令也用于切换分支和历史回退，所以这样做能够避免歧义

如果改动提交以后，后悔了怎么办？提交的行为已经被记录成了历史节点，所以就只能“篡改”历史记录了。

倒退历史

使用git reset <commit_id>进行的撤销操作，会导致历史回退到之前的某个提交节点。和之前提到的“回到过去”不同的是，checkout只是移动HEAD指针指向某个历史提交位置，不会改变当前分支末端的位置，所以保留了分支里所有历史信息。而reset移动当前分支末端的位置到指定的提交位置，因此这里就成为历史最新的节点HEAD，该节点之后的所有提交就被“丢弃”了，结果就是该节点后所有的历史记录全部被清空（当然也不是立即被清空，git的垃圾回收会默认等待30天）。

reset命令有3个参数，会根据不同参数进行不同程度的回退效果：

  • --hard将回退工作区、缓冲区和提交历史区到指定的提交节点，来还原当时的情景，因此会丢弃当下工作区和缓冲区中的全部改动，也被视为危险操作
  • --mixed将回退缓冲区和提交历史区到指定的节点，因此在缓冲区下的改动会返回到工作区
  • --soft只回退提交历史区到指定的节点，多余的改动保留进了缓冲区，而工作区保持不变

git reset就是前面提到在正式提交前，进行后悔的撤销操作，其实该命令是git reset --mixed HEAD的简化版，即撤销到最近一次的提交历史。如果通过reset回退到以前，再想重新回到现在的话，还可以通过git reflog查找最新执行过的git操作，定位到回退操作前所在当前的commit_id，再用git reset --hard <commit_id>回来。

合作分支的历史修改

上面提到的所有关于历史的修改，仅仅限于本地操作，如果要修改一个多人合作分支的历史，就不建议这么做了，因为任何对过去历史节点的改动，都会造成与别人本地分支的不一致，无法顺利合作。但是这并不意味着在合作分支上就一定不允许“撤销”。git的revert命令就正好使用在这种场景，因为它所做的“撤销”不是删除历史节点，而是在当前的节点上增加一个新的节点来重新“撤销”后的结果。由于这种方式不会修改过去的历史记录，因此被认为是安全的“撤销”，这样就适合使用在多人合作的分支上。

合并

Merge or Rebase

关于合并操作，除了可以常用的merge以外，还有rebase。后者是将一系列提交移动或整合到一个新的提交的基础上，说白话就是“拷贝嫁接”。

这两个命令其实都是为了将分岔的工作流统一起来的方法，只是实现方式不同而已。merge是一种不会破坏历史线的安全方式，但是缺点是多次的合并会让不同的分支之间出现多次的穿插记录；而rebase是通过修改历史来创造出一个干净的线性工作流，方便代码审核与历史的可读性。

在合并或同步工作流时，如果害怕出错的话，完全可以使用保守的merge方式；而其他某些情况下是rebase施展身手的时刻，比如：在本地feature分支上重新整理之前的提交历史，构造一个更简洁清晰的历史线。首先用merge-base命令找到开启分支/分岔点，然后通过交互式的rebase来重头整理这个feature分支的每个提交。

但是使用rebase要遵守一条黄金定律，就是rebase不能用在公共分支上！因为其修改历史记录的特性会影响到其他协作同伴的工作流。

Pull with Rebase

如果需要从远程master分支获取项目新的改动并整合到本地，通常的操作是git pull origin master，这里的pull命令实际上做了两件事：通过fetch获取远程分支，并将其merge合并到本地分支，所以它是一个fetch + merge的组合操作。
